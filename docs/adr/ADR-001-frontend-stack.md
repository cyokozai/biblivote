# ADR-001: フロントエンドスタック選定

**ステータス**: Accepted
**日付**: 2026-03-27
**決定者**: 井上裕介（cyokozai）
**レビュアー**: -

---

## コンテキスト

Biblivoteは静的ファイルとしてGitHub Pages / Cloudflare Pagesにホスティングするシングルページの投票フォームである。以下の制約がある。

- ビルドインフラを持たない（CI/CDパイプラインを最小化したい）
- 開発者1名、開発期間3週間
- 投票フォームの寿命は約1ヶ月（2026-04-15〜05-15）
- バンドルサイズを最小限にしてモバイルのロードを高速化したい
- TypeScript/Reactのビルドステップを避けてデプロイを簡素化したい

## 検討した選択肢

### 選択肢 A: Vanilla JS（ビルドなし）
**概要**: HTMLファイルにインラインまたは`<script>`タグでJSを記述。Alpine.jsをCDNで読み込む選択肢も含む。
**メリット**:
- ビルドステップなし → `git push`だけでデプロイ完了
- 依存関係ゼロ（Alpine.jsはCDN）
- ページロードが最速
- 開発者1名のプロジェクトに適した複雑さ
**デメリット**:
- 画面数が増えるとDOM操作が煩雑になる
- TypeScriptの恩恵（型安全）がない
**コスト概算**: 初期コスト$0 / ランニングコスト$0

### 選択肢 B: React + Vite（TypeScript）
**概要**: Viteでビルドし、GitHub Pages / Cloudflare Pagesにデプロイする。
**メリット**:
- コンポーネント分割でウィザードフォームを整理しやすい
- TypeScriptによる型安全
- React Hooksでフォーム状態管理が明快
**デメリット**:
- ビルドステップが必要（CI/CD設定が増える）
- node_modulesの管理が必要
- バンドルサイズがVanillaより大きい（gzip後50KB+）
- 1ヶ月限定プロジェクトに対して過剰な設定コスト
**コスト概算**: 初期設定2〜4時間 / ランニングコスト$0

### 選択肢 C: Next.js（SSG）
**概要**: Next.js の `next export` で静的ファイル生成。
**メリット**:
- ファイルベースルーティングが便利
**デメリット**:
- 静的フォームに対して明らかに過剰
- Cloudflare Pages / GitHub Pagesの設定が煩雑
**コスト概算**: 初期設定4〜8時間

## 決定

**選択肢 A（Vanilla JS + Alpine.js CDN）を採用する。**

### 理由

プロジェクトのライフタイムが1ヶ月、開発者1名、画面数8画面というスコープにおいて、ビルドステップを持つフレームワークを導入するメリットよりもコストが上回る。Alpine.jsを使うことでリアクティブなデータバインディングを最小限の記述量で実現できる。ウィザードの状態管理は8画面分のシンプルなオブジェクトで十分。

### 前提条件

- 画面数が8以上に増えた場合は再検討する
- Alpine.jsのCDNが利用できることを前提とする（CDNダウン時の代替は考慮しない）

## 影響・結果

**ポジティブな影響**:
- デプロイが `git push` だけで完了する
- ビルド失敗のリスクがない
- ページロードが最速

**ネガティブな影響・受け入れるリスク**:
- 型安全性がない（バリデーション関数のユニットテストで補完する）
- コード補完が弱い（VSCodeのJSDoc型アノテーションで一部補完）
