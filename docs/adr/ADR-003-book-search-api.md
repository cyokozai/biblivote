# ADR-003: 書籍検索APIの選定

**ステータス**: Accepted
**日付**: 2026-03-27
**決定者**: 井上裕介（cyokozai）
**レビュアー**: -

---

## コンテキスト

Q4・Q5で「書名からの書籍検索サジェスト」機能が必要。要件は以下の通り。

- タイトル文字列での検索（部分一致）
- 書影（サムネイル画像）の取得
- APIキーなしで利用開始できること（または低コスト）
- 日本語の技術書を対象とする

## 検討した選択肢

### 選択肢 A: Google Books API
**概要**: GoogleのPublic Books API。タイトル検索と書影を1エンドポイントで取得可能。
**メリット**:
- タイトル検索（`q=intitle:`）が可能
- 書影URLを `imageLinks.smallThumbnail` で取得できる
- APIキー不要で即時利用開始可能
- ISBNも取得可能（`industryIdentifiers`）
- 日本語書籍に `langRestrict=ja` フィルタが使える
**デメリット**:
- 書影URLが `http://` で返るため `https://` に変換が必要
- 無料レート制限あり（APIキーなし: 1000リクエスト/日）
- データ品質は完全ではない（古い書籍や同人誌は情報が薄い）
**コスト概算**: $0（レート制限内）

### 選択肢 B: openBD
**概要**: 日本の出版業界団体が提供する書籍情報API。
**メリット**:
- 日本の書籍に特化
**デメリット**:
- **ISBN指定のみ**でタイトル検索が不可能（致命的な欠点）
- 2023年のJPROデータ配信停止により書影が大幅欠損（Kubernetes完全ガイド、Real World HTTP等で検証済み → 全て空）
**コスト概算**: $0

### 選択肢 C: 楽天ブックスAPI
**概要**: 楽天APIのbooks検索。タイトル検索と書影が取得可能。
**メリット**:
- 日本の書籍データが充実
- タイトル検索が可能
**デメリット**:
- APIキー登録が必要（楽天デベロッパーアカウント）
- CORS制限があり直接ブラウザからの呼び出しが不可（プロキシが必要）
**コスト概算**: $0、プロキシ実装コスト+4時間

## 決定

**選択肢 A（Google Books API）を採用する。**

### 理由

openBDはタイトル検索不可と書影欠損という2つの致命的欠点があり採用不可。楽天ブックスAPIはCORS制限でGASプロキシが必要になり実装コストが増加する。Google Books APIはタイトル検索・書影・ISBNを1リクエストで取得でき、APIキー不要で即時利用開始できる。1000リクエスト/日の制限は「投票者数 × デバウンス後の検索回数」で概算すると200票 × 平均5回 = 1000リクエストとなり上限ギリギリ。GCPでAPIキーを取得すれば上限引き上げが可能。

### 前提条件

- 1000リクエスト/日の制限を超える見込みになった場合、GCPでAPIキーを取得する
- 書影URLの `http://` → `https://` 変換を必ず実装する（CSP違反を防ぐため）

## 影響・結果

**ポジティブな影響**:
- フロントエンドから直接APIを呼び出せる（プロキシ不要）
- 即時利用開始可能

**ネガティブな影響・受け入れるリスク**:
- レート制限超過のリスク（GCPキーで対応）
- データ品質に依存（古い書籍の書影なし等は許容）
- Googleのサービス変更・廃止リスク（イベント期間1ヶ月なので許容）
